# 1.0 发布契约

`api/release-contract-v1.json`是言界 1.0 的机器可读发布边界。它把分散在 API 快照、历史
兼容报告、供应链报告、工作流和文档中的稳定承诺汇总成一个可重建对象；CI 每次重新生成并
逐字节比较。

## 冻结内容

- 公开 API 摘要及 30 个声明、23 个类、7 个函数、125 个域、375 个方法的数量；
- `v0.1.0`至`v0.9.0`十一个已发布基线全部兼容，以及兼容报告自身的 SHA-256；
- 言台 1.0.0、ABI v2、言据 1.1.2 的精确提交和供应链报告摘要；
- 平台 1.7、事件 1.3、无障碍 1.0、绘制 1.1 的消费边界；
- 言序 1.1.9 最低工具链、1.1.20 Release 工具链和六个正式桌面目标；
- 14 项单元测试、11 条无窗口集成、5 条真实窗口路径、15 个公开示例、每目标 16 个 YXB、
  4096 轮交互和 1024 轮资源归零门禁；
- 四个公开 Release 附件的命名模式、支持政策摘要和两份许可证摘要。

契约不复制全部 API 或依赖记录；它用摘要引用对应权威 JSON。任一权威文件变化都必须显式
重建契约，并在变更提交中说明兼容性。1.x 新增可选能力时可以扩展契约；删除字段、降低门禁
数量或改变既有协议边界必须按主版本变更处理。

## 本地重建

从仓库根目录运行：

```sh
node scripts/check-release-contract.mjs --root . --report /tmp/release-contract-v1.json
cmp api/release-contract-v1.json /tmp/release-contract-v1.json
```

生成器同时验证 API 与冻结快照逐字节一致、兼容基线顺序、上游精确提交、原生库边界、测试
与示例数量、CI 工具链／目标标记、稳定文档和`SUPPORT.md`关键章节。报告不能靠手工改 JSON
绕过这些来源检查。

## 候选与 Release

六目标候选归档必须携带发布契约。候选演练会把归档内文件与标签提交逐字节比较，并把契约
摘要写入`release-candidate-report`；正式 Release 又核对该报告的来源提交和归档摘要。
因此公开附件、锁、API、兼容、供应链和稳定承诺都来自同一个已经通过标签 CI 的提交。

发布契约不替代应用自己的目标系统验收。字体、输入法、屏幕阅读器、系统主题和桌面服务仍
需在应用实际部署环境中验证，完整清单见[生产验收](PRODUCTION_VALIDATION.md)。

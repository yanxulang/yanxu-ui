# 0.x–1.0 API 兼容证明

言界 1.0 把“旧程序仍可编译”从文字承诺改为可执行门禁。CI 从每个已发布标签读取当时的
`api/api-v1.json`，逐一与当前 1.0 候选比较；任一旧声明、域或方法消失，参数收窄，静态／
异步属性改变，父类改变或返回类型扩张都会失败。

## 已验证基线

当前报告覆盖以下十一个已发布基线：

| 阶段 | 标签 |
| --- | --- |
| 初始公开面 | `v0.1.0`、`v0.1.1`、`v0.1.2` |
| 主题与原生无障碍 | `v0.2.0` |
| 完整表单 | `v0.3.0` |
| 叠层与菜单 | `v0.4.0` |
| 文本与数据视图 | `v0.5.0` |
| 动画与诊断 | `v0.6.0` |
| 生命周期与恢复 | `v0.7.0` |
| 跨平台生产验收 | `v0.8.0` |
| 1.0 候选冻结 | `v0.9.0` |

机器报告见[`api/compatibility-v1.json`](../api/compatibility-v1.json)。1.0 稳定公开面包含
30 个声明、23 个类、7 个包级函数、125 个域和 375 个方法；当前 API 与冻结快照的
SHA-256 均为`60596fd0f1c6e4a89cb44acd6109fe40945d386fa617d8f562ee8e8486c1a228`。

## 判定规则

- 顶层声明必须保留相同名称和种类；类必须保留原父类与协议能力；
- 旧域可以由当前类或其父类提供，但类型、只读和静态属性不能改变；
- 旧方法可以移动到父类，但参数名称、顺序、类型、静态和异步属性不能改变；
- 返回类型可以保持不变，或从旧`任意`／联合类型安全收窄；不能增加旧调用方未声明处理的
  新类型；
- 新声明在 0.8 以前允许作为向后兼容扩展；`api/api-v1.freeze.json`建立后，0.9 又要求
  当前生成 API 与候选冻结逐字节一致，因此不再新增公开表面。

继承查找是证明的一部分。例如弹出层曾把默认事件实现移到父类；旧代码仍能调用该方法，
不能把“子类快照不再重复列出继承方法”误判为删除。

## CI 证据

策略任务使用完整 Git 历史，依次执行：

1. 从十一个标签提取 API JSON；
2. 运行[`scripts/check-api-compat.mjs`](../scripts/check-api-compat.mjs)；
3. 重新生成兼容报告并与仓库报告逐字节比较；
4. 用最低言序重新生成当前 API 和参考文档；
5. 同时比较当前快照、1.0 候选冻结和生成结果。

兼容脚本另有负向验收：从当前 API 删除一个旧包级函数时必须以非零状态退出。参数和返回
规则只证明源码签名兼容；资源所有权、事件来源和无障碍树结构等行为迁移见
[0.x 升级到 1.0](MIGRATION_0_X_TO_1_0.md)。

## 边界

本证明不承诺序列化应用跨版本永久复用；生产应用应使用目标版本重新生成锁并构建 YXB。
它也不把 GPU、移动端或 Web 纳入 1.0。平台 ABI 和四套协议的兼容证明由言台 1.0 提供，
言界在六目标真实窗口中验证其消费结果。
